# 圆桌个人知识库设计

## 1. 目标与边界

在 `library/` 下建立一个由 Obsidian CLI 维护的个人知识库，用于持续积累：

1. 经筛选、未来可能复用的研究材料；
2. 圆桌或其他讨论中形成的可复用结论；
3. 材料、结论与既有领域知识之间的联系。

知识库以领域知识为中心，不按讨论场次组织主体目录。首期领域为 A 股投资、大宗商品和上海房产。

该知识库是用户的个人笔记，不是嘉宾 Agent 的默认共享上下文。任何工作流是否读取或写入知识库，由外部调用方决定。

## 2. 设计原则

### 2.1 领域中心

知识页按稳定领域、对象和讨论方向组织。同一主题反复更新，不因一次搜索或一次讨论新建平行知识页。

### 2.2 最小必要

- 优先更新已有页面，避免无脑追加文件、段落和标签。
- 只有未来可能复用的材料才归档。
- 只有形成可复用认识的讨论才创建结论记录。
- 不保存完整搜索流水、未采用结果或讨论逐字稿。
- 元数据只保留检索、判断可信度和追溯所需字段。

### 2.3 当前最佳认知

领域知识页表达当前最佳认知，可在新证据出现后重写、合并或删除旧表述。讨论结论记录保留当时的判断，提供追溯依据。

### 2.4 重大变更门禁

以下变更不得自动覆盖，必须生成待确认项：

- 核心结论反转；
- 置信度跨级变化；
- 删除仍有依据的有效内容；
- 可靠来源之间出现实质冲突。

普通事实补充、措辞压缩、链接完善和同方向证据增强可自动完成。

### 2.5 能力解耦

知识库不定义主持人、嘉宾、调研员或记录员，也不定义圆桌顺序。它只接受两类调用方：

- 材料提供者：提交候选材料；
- 结论提供者：提交讨论或分析结论。

调用方可以是任何 Agent、人或未来工作流。角色分配与调度由主持 Skill 或其他上层编排负责。

## 4. 内容模型

### 4.1 领域知识页

领域知识页是主要读取入口，围绕一个稳定主题维护当前认知。建议仅包含：

- 当前判断；
- 关键机制或因果链；
- 重要证据与反证；
- 适用范围与条件；
- 未解决问题；
- 相关来源和结论记录链接。

写入前必须先搜索同义主题和相关页面。若已有页面能够容纳新内容，则合并而不是新建。

### 4.2 精选来源

来源只有在满足至少一个条件时归档：

- 支撑或挑战一个重要判断；
- 提供可重复使用的数据、机制或验证路径；
- 后续讨论很可能再次引用。

来源记录仅保留：标题、URL 或出处、发布日期、访问日期、证据等级、极简摘要、采用原因、关联知识页。相同 URL、文档或实质相同材料只保留一份；再次使用时补充链接，不重复建页。

原文是否本地保存按版权、可访问性和复用需要决定，不默认复制全文。

### 4.3 结论记录

结论记录用于保留某次讨论在当时形成的有效判断，不承担领域知识组织职责。仅在存在可复用结论时创建，包含：

- 日期与讨论主题；
- 简短结论；
- 成立条件和置信度；
- 关键依据；
- 分歧或未决问题；
- 已回写的领域知识页。

完整对话、角色发言过程和无结论讨论不进入知识库。

## 5. 写入流程

### 5.1 材料整合

1. 使用 Obsidian CLI 搜索相同来源和相关知识页。
2. 判断材料是否达到精选标准；未达到则不写入。
3. 对来源去重，创建或更新一份来源记录。
4. 将必要信息整合到已有知识页；只有缺少合适承载页时才新建。
5. 补充双向链接和领域索引。
6. 若触发重大变更，将提案写入待确认文件，不改动当前结论。
7. 返回本次创建、更新、链接、跳过和待确认项。

### 5.2 结论整合

1. 判断结论是否可复用；纯场次摘要不写入。
2. 创建一份简短结论记录。
3. 搜索并更新相关领域知识页，压缩重复表述。
4. 链接支撑来源和被更新的知识页。
5. 若触发重大变更，保留结论记录并生成待确认提案，暂不覆盖领域页。
6. 返回本次创建、更新、链接和待确认项。

### 5.3 人工确认

待确认项应展示：现有判断、拟议判断、变更原因、证据、影响页面和建议动作。确认后再通过 Obsidian CLI 更新领域页，并从待确认清单移除或标记完成。

## 6. Skill 设计

首期只创建两个能力型 Skill，避免过度拆分：

### 6.1 `library-integrate`

统一处理材料和结论输入，通过 `input_type: source | conclusion` 区分。两类流程共享搜索、去重、页面选择、整合、链接和重大变更判断，因此不拆成两个重复 Skill。

Skill 只包含：

- 触发条件；
- 最小输入要求；
- 整合优先和重大变更规则；
- Obsidian CLI 操作顺序；
- 简洁的结构化返回格式。

不包含领域知识、圆桌角色说明或主持调度逻辑。

### 6.2 `library-maintain`

用于按需检查重复页面、孤立页面、失效链接、未解析链接、冲突、过度碎片化和待确认项。它不参与日常写入，避免每次整合加载维护规则。

### 6.3 Skill 最小化约束

- 每个 Skill 聚焦一种能力边界，不按 Agent 名称创建 Skill。
- 能共享的写入逻辑合并，不复制流程。
- SKILL.md 只保留执行所需规则；模板放入 `library/模板/`，不在 Skill 中重复。
- 优先调用 `obsidian help <command>` 获取命令细节，不在 Skill 中罗列完整 CLI 手册。
- 没有重复、脆弱的机械操作前，不增加脚本。
- 没有实际需求前，不增加额外 references、assets 或辅助文档。

## 7. 项目规则文件

采用三层职责：

- `AGENTS.md`：项目级不可违反的知识库原则、目录边界、Obsidian CLI 要求和重大变更门禁。
- `CLAUDE.md`：Claude Code 的轻量入口，要求遵守 `AGENTS.md`，不复制整套规则。
- Skill：按需加载的具体执行流程。

规则文件不描述主持工作流。主持 Skill 只需在合适时机调用知识库能力，并提供符合契约的输入。

## 8. Obsidian CLI 约束

- 所有知识库读取、搜索、新建、追加、重命名、链接检查均使用 `obsidian` CLI。
- 命令显式指定 `vault=everything-roundtable`，减少活动 vault 变化造成的误操作。
- 写入前使用 `search`、`files` 或 `read` 查重和定位。
- 写入后使用 `links`、`backlinks`、`unresolved`、`orphans` 等命令做必要验证。
- 不使用 CLI 永久删除；需要删除时先进入人工确认流程。

## 9. 首期验收标准

- `library/` 具备三个领域入口、来源层、结论记录层、模板和维护入口。
- 任意材料提供者可以提交精选来源并得到去重、整合后的结果。
- 任意结论提供者可以提交结论并完成结论记录与领域页回写。
- 普通更新可以自动完成，重大变更会停止覆盖并生成清晰的确认项。
- Skill 不依赖固定圆桌角色，也不承担调度职责。
- 知识页没有按讨论场次扩散，重复内容会被合并或跳过。
- Obsidian CLI 能列出、读取、搜索和检查新建知识库文件。

## 10. 暂不实现

首期不引入 Dataview、向量检索、全文抓取、自动网页快照、复杂脚本、数据库、嘉宾知识库同步或自动主持编排。只有实际规模和使用反馈证明必要时再增加。
